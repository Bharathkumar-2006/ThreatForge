============================================================
THREAT MODELING REPORT: Mailcheckr
============================================================
Description: MailCheckr is a browser-based email security analysis platform that helps users evaluate sender reputation, fraud risk, and email trustworthiness. The system analyzes email metadata and integrates with external reputation services to provide security insights and phishing detection assistance.

Tech Stack: Chrome Extension, JavaScript, Node.js, Express.js, MongoDB, REST APIs, JWT Authentication

Notes: Scope:
- User authentication and dashboard access
- Email sender analysis
- Reputation and fraud-risk assessment
- Integration with third-party reputation services
- Browser extension and backend communication

Security Notes:
- JWT-based authentication
- Input validation on all API endpoints
- Secure handling of external API keys
- Protection against brute-force and credential stuffing attacks
- Secure storage of user data
- Rate limiting on sensitive endpoints
- Prevention of information disclosure through error messages

============================================================
Threats Summary (6 threats identified):
============================================================

Module: Sign in
----------------------------------------
Threat Name: Credential Brute‑Force via API Endpoint
STRIDE:      Denial of Service / Elevation of Privilege
Risk Level:  High
Scenario:    An attacker scripts rapid POST requests to /auth/login with guessed password combinations for a target email address. Because the endpoint lacks aggressive rate‑limiting per IP/account, the attacker can eventually discover a valid password or lock out the legitimate user, causing a temporary denial of service and potential account takeover.
Mitigation:  Implement exponential back‑off and CAPTCHAs after a configurable number of failed attempts per IP and per account. Enforce account lockout after a threshold and notify the user. Use adaptive authentication (e.g., device fingerprint, geolocation) to add friction for suspicious login patterns.

Threat Name: JWT Manipulation (Tampering)
STRIDE:      Tampering / Elevation of Privilege
Risk Level:  Critical
Scenario:    An attacker intercepts a valid JWT (e.g., via XSS in the extension) and modifies the payload to upgrade the "role" claim from "user" to "admin". If the backend does not verify the token signature against a strong, rotating secret or public key, the altered token is accepted, granting privileged access.
Mitigation:  Sign JWTs with a strong HS256/RS256 secret stored securely (e.g., env var, secret manager) and rotate keys regularly. Validate the signature on every request, enforce strict claim validation (issuer, audience, expiration). Use short‑lived access tokens and refresh tokens stored httpOnly, Secure, SameSite=Strict.

Threat Name: User Enumeration via Timing Differences
STRIDE:      Information Disclosure
Risk Level:  Medium
Scenario:    The login API responds faster when the supplied email does not exist versus when it exists (because password hash verification is skipped). An attacker measures response times across many attempts to compile a list of valid email addresses for further phishing or credential stuffing.
Mitigation:  Normalize response time for both success and failure cases (e.g., always perform a dummy password hash verification). Return a generic error message and status code regardless of account existence.

Threat Name: Replay Attack on JWT after Logout
STRIDE:      Spoofing / Repudiation
Risk Level:  Medium
Scenario:    After a user logs out, the JWT remains valid until its expiration. An attacker who captured the token (e.g., via compromised browser extension storage) can replay it to access protected endpoints, impersonating the user.
Mitigation:  Implement token revocation lists or use short‑lived access tokens (e.g., 5‑15 minutes) with refresh tokens stored server‑side. Invalidate tokens on logout by adding them to a blacklist cache with TTL matching the token's remaining life.

Threat Name: Cross‑Site Scripting (XSS) in Extension UI Leading to Credential Theft
STRIDE:      Tampering / Information Disclosure
Risk Level:  Critical
Scenario:    A malicious third‑party script is injected into the extension's popup (e.g., via a compromised CDN). The script reads the user's email/password fields and exfiltrates them to an attacker-controlled server before submission, compromising credentials.
Mitigation:  Apply a strict Content Security Policy (CSP) in the extension manifest, avoid dynamic `eval` or `innerHTML`, and bundle all scripts locally. Perform input sanitization and use Subresource Integrity (SRI) if external resources are unavoidable.

Threat Name: Insufficient Logging Sanitization (Log Injection)
STRIDE:      Tampering / Repudiation
Risk Level:  Low
Scenario:    An attacker includes newline characters or special log delimiters in the email field during login. If the backend logs raw input without sanitization, the log file can be corrupted, potentially masking audit trails or enabling log‑forging attacks.
Mitigation:  Sanitize or encode all user‑supplied data before writing to logs. Use structured logging (JSON) where fields are separated by schema, and enforce length limits on logged values.
